# Host Adapters

CDF is becoming a host-adaptable framework. The core methods should work in Claude Code, Codex, and other coding-agent runtimes without rewriting every command.

## Architecture

Keep a hard split between core methodology and host adapters.

| Layer | Owns | Examples |
|-------|------|----------|
| Core methodology | What the agent should do | `commands/`, `agents/`, `skills/`, `rules-templates/`, `docs/` |
| Host adapter | How the host loads and invokes it | `.claude-plugin/`, `CLAUDE.md`, `AGENTS.md`, host skill install paths, lifecycle hooks |
| Machine config | Local credentials and integrations | MCP settings, API keys, local logs, browser state |

Core files should avoid host-only claims such as "Claude tool" or "Task tool" unless the behavior only exists in the Claude adapter. Prefer "subagent", "host skill", "command prompt", or "lifecycle hook".

## Adapter Contract

Every supported host needs five capabilities:

1. **Instruction loading** - Put the role, rules, routing table, and critical commands into the host's always-loaded project file.
2. **Command invocation** - Expose `/cdf:*` methods as slash commands, prompt aliases, or documented natural-language triggers.
3. **Skill routing** - Load CDF skills when request context matches their triggers.
4. **Lifecycle checks** - Run validation hooks if the host supports hooks; otherwise expose equivalent local scripts.
5. **Verification** - Run `python3 scripts/health-check.py` after adapter or routing changes.

## Claude Code Adapter

Current packaged adapter:
- `.claude-plugin/plugin.json` and `.claude-plugin/marketplace.json`
- `commands/*.md` exposed as `/cdf:*`
- `skills/*/SKILL.md` auto-invoked by Claude Code
- `.claude/rules/` generated by `/cdf:rules generate`
- `hooks/hooks.json` lifecycle hooks
- `CLAUDE.md` / `CLAUDE.generated.md` project instructions

Claude-specific behavior belongs in these adapter files or in clearly marked sections.

## Codex Adapter

Codex support should use the same core methodology with Codex-native loading:
- `AGENTS.md` for project role, rules, routing, and command selection
- Codex skills installed under the Codex skill path, ideally prefixed as `cdf-*`
- Natural-language command prompts that preserve `/cdf:*` names even if the host does not provide slash-command dispatch
- `scripts/health-check.py` as the portable validation gate

Initial Codex routing should map:

| Intent | CDF route |
|--------|-----------|
| Implement a feature | `/cdf:implement` prompt |
| Debug a failure | `/cdf:troubleshoot` prompt |
| Challenge a plan | `/cdf:plan-review` prompt |
| Run quality checks | `/cdf:verify --mode pre-pr` prompt |
| Multi-file investigation | `/cdf:task` with codebase-navigator |
| Commit or ship | `/cdf:git` or `/cdf:ship` prompt |

Do not claim Codex has Claude Code lifecycle hooks. If a Codex host lacks hooks, route users to the equivalent command:

```bash
python3 scripts/health-check.py
python3 scripts/analyze-codebase.py
```

## Migration Rules

- Add new capabilities to the core first, then adapter glue second.
- Keep command names stable across hosts: `/cdf:plan-review` means the same review gauntlet everywhere.
- Prefer one strong skill over several persona-stub agents.
- Treat host-specific automation as optional acceleration, not required semantics.
- Update this file when adding a new host adapter.

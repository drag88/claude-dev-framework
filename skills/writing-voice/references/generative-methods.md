# Generative Methods

These are thinking tools for finding the right angle and structure before writing. Choose one method per piece. The editorial pipeline (editorial-pipeline.md) refines what these methods produce.

## Method 1: Micro Story (Default)

The mind is wired for stories. Use transformation as the spine. This is the default starting point — most ideas can be expressed this way.

**Structure:**
1. **Problem** — State a relatable problem observed or experienced
2. **Amplify** — Illustrate the negative outcome if unsolved
3. **Solution** — Deliver the insight or approach

**When to use:** Most topics. Especially when you have a specific experience that illustrates the point.

**Example seed:** "Nobody uses our dashboards" → Problem: built dashboards nobody uses. Amplify: three months of work, zero decisions changed. Solution: the issue was not the dashboard — it was that nobody asked what decision it was supposed to enable.

## Method 2: Pyramid Principle (Building Depth)

For ideas requiring structured support. Answer-first, evidence-after.

**Structure:**
1. **Lead with the conclusion** — State the main idea first
2. **Support with 3-5 key arguments** — Ask "why" repeatedly to generate these
3. **Ground with evidence** — Personal experience, data, examples from others

**When to use:** When the insight needs building. When the reader might not accept the conclusion without the supporting logic. Longer LinkedIn posts and blog sections.

**Example seed:** "Generalists outperform specialists in leadership" → Conclusion stated. Why? (1) Translation ability across domains. (2) Pattern recognition from diverse experience. (3) Adaptability to new problems. Evidence for each from career experience.

## Method 3: Cross-Domain Synthesis (Most Distinctive)

The most memorable content connects ideas across disciplines. This is Aswin's strongest mode — bridging data/tech with leadership/human insight.

**Structure:**
1. **Problem and amplify** — Hook with a relatable problem
2. **Cross-domain insight** — Pull a concept from another field (physics, psychology, economics, sports, music) to illuminate the topic
3. **Unique solution** — Offer a perspective that only emerges from this synthesis

**When to use:** When a technical concept has broader implications. When you notice a parallel between your domain and another. When you want to reach both technical and non-technical audiences.

**Example seed:** "Data quality is entropy" → Problem: data quality degrades over time. Cross-domain: entropy from physics — all systems tend toward disorder without energy input. Solution: quality is not a state you achieve, it is a fight against entropy you never stop fighting.

**Technical domains to mine for philosophical insight:**
- Optimization (gradient descent, local vs global minima, loss functions)
- Systems design (distributed systems, fault tolerance, redundancy)
- Information theory (entropy, signal vs noise, compression)
- Database concepts (transactions, consistency, eventual consistency)
- Software patterns (technical debt, abstraction, coupling)
- ML fundamentals (overfitting, regularization, bias-variance tradeoff)

## Method 4: Compression (Tweets Only)

For single tweets where every character matters. This is not "write long then cut" — it is finding the irreducible core of an idea.

**Process:**
1. State the idea in one sentence
2. Cut it in half. Is the meaning intact?
3. If yes, cut again. If no, you have found the essential words.
4. Check: does it read as discovered truth or manufactured cleverness? If manufactured, start over.

**Structures that work at tweet scale:**
- **Definition:** "X is Y" — clean, declarative. "Wealth is assets that earn while you sleep."
- **Contrast:** "Not X, but Y" — challenges assumptions. "Seek wealth, not money or status."
- **Observation:** Specific thing noticed + what it reveals. No preamble.
- **Reframe:** Common belief stated, then twisted. "We talk about 'data-driven decisions' like data does the deciding."
- **Pattern:** "Same pattern keeps emerging:" + the pattern.
- **Confession:** Vulnerability compressed to one sentence. No self-pity.

**When NOT to use Compression:** If the idea genuinely needs development (context, evidence, nuance), it is not a tweet. Use a thread or LinkedIn post instead. Do not compress ideas that need room to breathe.

## Choosing a Method

| Signal | Method |
|--------|--------|
| You have a specific story or experience | Micro Story |
| The conclusion needs logical support | Pyramid Principle |
| A technical concept has broader implications | Cross-Domain Synthesis |
| The idea fits in one sentence | Compression |
| You are not sure | Start with Micro Story, switch if it does not fit |

## Connecting to Core Beliefs

Before choosing a method, identify which core belief from voice-and-beliefs.md connects to the topic. This creates consistency of voice and depth that audiences recognize over time.

Not every piece needs a core belief anchor. But the best ones do — they feel like part of a larger conversation rather than isolated takes.
